# 01 — Project Briefing Protocol

## Purpose

Transform the Raw Briefing into a structured Project Briefing.

This is the first product artifact generated by MAWA.
It exists before Discovery and acts as the bridge between a rough idea and architectural analysis.

## Inputs

Primary input:

- `inputs/raw-briefing.md`

Also use:

- human clarifications gathered during the briefing interview (see Required process — this is part of the protocol, not an optional extra);
- existing `.mawa-config.yaml`, if present.

## Output

- `specs/briefing/project-briefing.md`

## Required template

Use `resources/templates/project-briefing.template.md` as the required output structure for `specs/briefing/project-briefing.md`.

Do not generate the Project Briefing in a free-form structure unless the human explicitly overrides the template.

## Role

Act as a Senior Product Strategist and Principal Solution Architect.

## Important distinction

Do not generate:

- discovery spec;
- domain map;
- database schema;
- implementation plan;
- code.

This stage only creates a structured Project Briefing.

The section list lives in `resources/templates/project-briefing.template.md` (the single
source of truth). Generate `specs/briefing/project-briefing.md` with every section from
that template, in order. Do not redefine or renumber the sections here.

## Required process — critical briefing interview

The Raw Briefing is a rough seed. This stage must actively **interrogate and expand** it,
not just reformat it into the template. Before writing the Project Briefing:

1. Read the Raw Briefing and build an internal model of the product.
2. Hunt for gaps, ambiguities, hidden assumptions, technical risks, missing edge cases,
   vague UX, integration unknowns, and unclear ownership of state/data.
3. Conduct an iterative interview with the human:
   - ask **one question at a time**, most critical first;
   - be critical — challenge weak, missing, or contradictory decisions instead of
     accepting them at face value;
   - maintain and revise your internal decision model as the human answers;
   - record each resolved point so it lands in the briefing.
4. Continue the loop until you have a **decision-complete** picture — enough to write a
   briefing that genuinely supports Discovery, not just a tidy restatement of the seed.
5. Only then write `specs/briefing/project-briefing.md`, capturing surfaced risks in the
   `Risks and Gaps` section and unresolved items in `Open Questions` / `Assumptions`.

Do not write the briefing while material, blocking ambiguities remain unresolved.

## Interaction rules

In `interactive` mode:

- run the critical interview above; ask one question at a time;
- do not batch questions or dump a questionnaire at once;
- non-blocking details the human cannot or need not answer become explicit
  `Assumptions` / `Open Questions` — but still surface the important ones;
- before writing the briefing, confirm readiness, e.g.:

  > I have a complete enough picture to write the Project Briefing. Ready for me to generate it?

In `continuous` mode:

- the interactive interview is not possible, so perform the same critical analysis
  internally and record every gap/risk as an explicit `Assumption`, `Open Question`, or
  `Risks and Gaps` entry;
- proceed with documented assumptions;
- stop only if no meaningful Project Briefing can be created.

## Definition of done

The briefing only needs to be good enough to support Discovery — it does not need to
eliminate every open question. Do not emit the completion message until all are true:

- [ ] The critical briefing interview (interactive) or critical gap analysis (continuous) was performed.
- [ ] Every section from `resources/templates/project-briefing.template.md` is present in `specs/briefing/project-briefing.md`.
- [ ] Surfaced risks, gaps, and ambiguities are captured (`Risks and Gaps` / `Open Questions` / `Assumptions`).
- [ ] Every blocking gap is resolved or recorded as an explicit assumption.
- [ ] The Discovery readiness assessment is stated.
- [ ] No discovery, domain, spec, or code content was generated.

## Stage completion message

End with:

> Project Briefing is ready at `specs/briefing/project-briefing.md`. May I proceed to Stage 02 — Environment Setup?
